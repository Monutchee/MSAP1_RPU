# AD7771 RPU interface

R5 core 0 owns the AD7771. The public C++ interface is
`common/include/ad7771.hpp`; applications do not access AXI registers or AD7771
register addresses directly.

## Default profile

- High-resolution mode
- Sinc5 filter
- 32 kSPS simultaneous sample rate
- 256 frames per DMA packet
- Eight signed, sign-extended 32-bit channel values per frame

The library also defines 1, 2, 4, 8, 16, 32, 64, and 128 kSPS profiles.
The sensor board supplies the AD7771 with a fixed 8.192 MHz MCLK, so this board
uses high-resolution mode; the AD7771 specifies a maximum 4.096 MHz MCLK in
low-power mode. The API still represents low-power mode for reuse with hardware
that provides a suitable clock, and rejects invalid clock/rate/decimation
combinations. The library keeps the four-lane data clock at MCLK and programs
the SRC decimation to select the sample rate.

The sensor-board schematic also labels the analog input low-pass sections at
approximately 12.5 kHz and identifies 32 kSPS as the board sample rate. Thus,
32 kSPS is a sensible first production profile (16 kHz Nyquist bandwidth).
Higher ODRs are supported for experiments and oversampling, but do not increase
the analog front-end bandwidth.

## Hardware contract

| Function | Address | Owner |
| --- | ---: | --- |
| AXI Quad SPI | `0xB0010000` | AD7771 register control |
| AD7771 capture registers | `0xB0020000` | PL receiver control/status |
| AXI DMA registers | `0xB0030000` | Simple S2MM transfer to DDR |

Vivado's implemented address map and exported XSA both place AXI Quad SPI at
`0xB0010000`. Vitis 2025.2 may flatten that IP inside the block-design
container as `0xA0000000` in its generated SDT. The library uses the generated
entry to obtain the `XSpi` capabilities, then passes `0xB0010000` as the
effective hardware address to `XSpi_CfgInitialize()`.

The PL emits one AXI4-Stream beat per channel in channel order 0 through 7.
Each beat is a sign-extended 24-bit sample in a 32-bit word. `TLAST` is asserted
after `frames_per_packet * 8` beats, so a 256-frame packet occupies 8192 bytes.

`SampleFrame` is the stable software representation:

```cpp
msap1::adc::SampleFrame frame;
auto channel_zero = frame.channel[0];
```

DMA buffers must be 64-byte aligned and contain at least the configured packet
size. The R5c0 application uses two static buffers: it re-arms one before it
processes the other. The PL block-RAM FIFO stores 512 complete frames, providing
4 ms of descriptor-gap tolerance at 128 kSPS and 16 ms at the default 32 kSPS.

## Startup sequence

`Ad7771::initialize()` performs the complete device sequence:

1. Initialize AXI Quad SPI in mode 0 and AXI DMA in simple S2MM mode.
2. Pulse the active-low AD7771 RESET input and wait for `INIT_COMPLETE`.
3. Configure power mode, sinc filter, four DOUT lanes, status headers, and the
   requested SRC decimation.
4. Toggle `SPI_SYNC`. The sensor board connects `SYNC_OUT` to `SYNC_IN`, so this
   resets and synchronizes the digital filters without changing registers.
5. Leave `START` low. AD7771 START is a positive synchronization pulse, not an
   active-low conversion enable; this design uses `SPI_SYNC` instead.

The first integration polls DMA completion from a FreeRTOS task. DMA interrupts
can replace polling later without changing the public sample format.

## Changing the rate

Change only the configuration passed to `initialize()`:

```cpp
msap1::adc::Configuration configuration;
configuration.sample_rate = msap1::adc::SampleRate::Sps64000;
configuration.filter = msap1::adc::Filter::Sinc5;
configuration.power_mode = msap1::adc::PowerMode::HighResolution;
configuration.master_clock_hz = 8192000;
```

Changing rate while capture is active is intentionally not part of the first
API. Stop capture, reinitialize with the new profile, then restart DMA.
